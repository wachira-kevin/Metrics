# ADR-001: Current Architecture Trade-offs and Proposed Improvements

- Status: Proposed
- Date: 2026-05-20
- Author: Kevin Wachira

## Context

The current implementation of the platform was designed to prioritize rapid development, simplicity, and ease of deployment. While this approach enabled faster iteration and delivery, several architectural limitations have emerged as the system scales in terms of traffic, security requirements, and analytics workloads.

This ADR documents the major trade-offs in the current implementation and proposes improvements for future iterations.

---

# 1. Authentication and Session Management Limitations

## Current Implementation

The current authentication implementation is lightweight and stateless, with limited session management capabilities.

### Trade-offs

#### Advantages
- Simpler implementation
- Reduced infrastructure complexity
- Faster development and deployment
- Lower operational overhead

#### Disadvantages
- No centralized session management
- Difficult to revoke tokens/sessions in real-time
- Limited support for concurrent session tracking
- Reduced visibility into active users/devices
- Potential security concerns for long-lived tokens
- Harder to implement features such as:
  - forced logout
  - session expiration policies
  - device management
  - token blacklisting

## Proposed Improvement

Introduce Redis-backed session management.

### Proposed Architecture
- Store active sessions in Redis
- Implement token/session invalidation
- Add refresh token rotation
- Enable centralized session tracking
- Support session expiry and revocation

### Benefits
- Improved authentication security
- Faster token validation
- Real-time session invalidation
- Better scalability for authentication workflows
- Enables advanced security features

### Trade-offs Introduced
- Additional infrastructure dependency (Redis)
- Increased operational complexity
- Need for Redis high availability and persistence strategy
- Additional maintenance and monitoring requirements

# 2. Monolithic Service Responsibilities

## Current Implementation

Authentication logic and metrics processing currently exist within the same service boundary.

## Trade-offs

#### Advantages
- Easier local development
- Simpler deployment pipeline
- Reduced inter-service communication
- Lower infrastructure costs initially
- Faster iteration for small teams

#### Disadvantages
- Tight coupling between authentication and metrics concerns
- Difficult to scale workloads independently
- Reduced fault isolation
- Larger deployment surface area
- Harder to maintain ownership boundaries
- Increased risk of service degradation affecting unrelated functionality

## Proposed Improvement

Adopt a microservices-oriented architecture by separating:
- Authentication Service
- Metrics Service

Use an identity provider such as Keycloak for authentication and authorization management.

### Proposed Architecture

#### Authentication Service
- Managed by Keycloak
- Handles:
  - user authentication
  - authorization
  - token issuance
  - identity federation
  - role-based access control (RBAC)

#### Metrics Service
- Dedicated service focused purely on:
  - metrics ingestion
  - aggregation
  - querying
  - reporting

### Benefits
- Independent scalability
- Clear separation of concerns
- Improved maintainability
- Better security practices
- Easier observability and monitoring
- Reduced coupling between domains
- Faster scaling of metrics-heavy workloads

### Trade-offs Introduced
- Increased operational complexity
- More infrastructure components to manage
- Network communication overhead
- Distributed tracing and debugging become more complex
- Requires service discovery and API gateway strategies

# 3. PostgreSQL as a Metrics Database

## Current Implementation

PostgreSQL is currently used for storing application metrics.

## Trade-offs

#### Advantages
- Familiar and mature ecosystem
- Strong consistency guarantees
- Easy setup and maintenance
- Good relational querying capabilities
- Suitable for low-to-medium metrics workloads

#### Disadvantages
- Not optimized for high-ingestion time-series workloads
- Aggregation queries become expensive at scale
- Slower write throughput for large metrics volumes
- Storage growth impacts query performance
- Index maintenance overhead increases significantly
- Time-series retention management becomes harder

## Proposed Improvement

Adopt a database optimized for metrics and time-series workloads.

### Candidate Options

#### InfluxDB
Optimized for:
- time-series data
- high write throughput
- retention policies
- metrics aggregation

##### Benefits
- Native time-series optimizations
- Efficient compression
- Built-in retention policies
- Faster aggregation queries

##### Trade-offs
- Additional database technology to maintain
- Query language differences
- Potential ecosystem learning curve

---

#### ClickHouse
Optimized for:
- analytical workloads
- high-volume event ingestion
- OLAP-style aggregations

##### Benefits
- Extremely fast analytical queries
- High compression rates
- Excellent scalability
- Efficient columnar storage

##### Trade-offs
- Operational complexity
- More infrastructure tuning required
- Different data modeling approach from relational systems

# Decision Summary

| Concern | Current Approach | Proposed Direction |
|---|---|---|
| Authentication | Stateless auth with limited session management | Redis-backed session management |
| Identity Management | Embedded in application | Externalized via Keycloak |
| Service Architecture | Shared responsibilities | Separate microservices |
| Metrics Storage | PostgreSQL | InfluxDB or ClickHouse |

---

# Consequences

## Positive Outcomes
- Improved scalability
- Better security posture
- More efficient metrics processing
- Clearer system boundaries
- Better long-term maintainability

## Negative Outcomes
- Increased infrastructure complexity
- Higher operational overhead
- Additional deployment and monitoring requirements
- Increased learning curve for the engineering team

---

# Future Considerations

- Introduce API Gateway for centralized routing and authentication
- Add distributed tracing (OpenTelemetry)
- Introduce Kafka or RabbitMQ for asynchronous metric ingestion
- Implement metrics retention and archival strategies
- Add horizontal scaling strategies for ingestion services
